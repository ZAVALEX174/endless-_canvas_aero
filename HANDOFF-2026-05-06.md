### ⚠ Браузерный кеш JS — главная боль

Скрипты в `index.html` подключены без cache-bust, и браузер агрессивно кеширует. После правки `*.js` обычный `window.location.reload()` через `preview_eval` **не подгружает свежий код** — ты будешь тестировать старую версию.

**Workaround в preview**:
```js
new Promise(resolve => {
  const s = document.createElement('script');
  s.src = '/js-for-index4/visualization.js?bust=' + Date.now();
  s.onload = resolve;
  document.head.appendChild(s);
})
```
Это переопределяет функции в IIFE-модуле, но НЕ переустанавливает `canvas.on()` / `addEventListener` хендлеры — они привязаны к старым реализациям. Для проверки логики обработчиков надёжнее **прямые вызовы** функции, которую правил, без участия браузерного listener'а.

**Юзеру**: после правок всегда нужен `Ctrl+Shift+R` (hard reload) — иначе видит старый код.

## 2. Слои — как сейчас работают (контекст п.7)

Подробное описание дано раньше в чате; краткое резюме:

### Структура
[layersManager.js](js-for-index4/layersManager.js) хранит:
```js
layers = [{ id, name, visible, locked, color, elevation }]
crossLayerConnections = [{ x, y, auto: bool }]
```

`default` — особый слой, нельзя удалить, всегда первый.

### Принадлежность объектов
Каждый объект имеет `properties.layerId`. По умолчанию = `getActiveLayerId()`.

### Изоляция в расчёте
[layersManager.js:592-598](js-for-index4/layersManager.js#L592) — `getCalcNodeKey(x, y, layerId)` возвращает:
- `"x_y@layerId"` если точка НЕ в `crossLayerConnections` → узлы разных слоёв в одной точке = РАЗНЫЕ узлы графа.
- `"x_y"` если точка cross-layer → ОБЩИЙ узел.

### Авто-детект cross-layer (п.7.2 — ✅)
[layersManager.js:533-621](js-for-index4/layersManager.js#L533) — `recomputeAutoCrossLayerConnections()`. Запускается debounced (200мс) после object:added/modified/removed для линий. Для каждой пары линий разных слоёв проверяет: лежит ли endpoint одной линии на другой (≤10px). Если да — создаёт `{ x, y, auto: true }` и разрезает целевую линию в этой точке через `_splitLinesAtCrossPoint`.

**НЕ удаляет** старые точки автоматически (q2 от заказчика — «до явной правки»).

### Цвета маркеров
🟢 Зелёный (`rgba(46,213,115,0.85)`) — ручная (`auto: false`, legacy).
🟡 Жёлтый (`rgba(240,165,0,0.9)`) — авто (`auto: true`).

Маркер кликабелен — клик удаляет точку. Обработчики через `mousedown`/`mouseup` с порогом 400мс для отличия клика от drag (хотя drag заблокирован lockMovement).

### Скрытая ручная кнопка
Кнопка «Связь слоёв» (`#crossLayerBtn`) скрыта `style="display:none"` ([index.html:71](index.html#L71)), q1 от заказчика — «пока скроем».

### 5.2 Известные ограничения / спорные моменты

#### Странные Q при «фан в середине открытой линии»
Заказчик показал (2026-05-06): фан в 1/3 длины линии, оба конца открытые → 66.667 / 33.333 (сумма = Q фана = 100). Заказчик считает это странным — ожидает uniform Q=100 по всей линии.

**Объяснение** (см. лог чата): в нашей модели degree-1 endpoint без `Атмосферы` и без `sealed`-флага трактуется как **неявная атмосфера** (P=0). Фан в середине → давление в фане выше нуля → расход разделяется обратно пропорционально сопротивлениям сегментов: `Q_left × R_left = Q_right × R_right`. При фане в 1/3 длины → 2:1.

**Workaround для заказчика**: один из концов нужно **запечатать** (правый клик по синему endpoint-маркеру → «Запечатать тупик»). Тогда весь Q идёт через другой конец → uniform 100 на всей линии.

**Если заказчик хочет другую интерпретацию** — это изменение математики солвера (защищённой зоны). Требуется явное обсуждение.

#### applyFlowColoring для реверс-only сетей не красит ничего
В `applyFlowColoring` ([visualization.js:367-475](js-for-index4/visualization.js#L367)) BFS идёт от sources (фанов с `isFlowSource !== false`) по `outEdges`. У реверс-фана после `writeFlowResults` нет `outEdges` (всё течёт В фан) → BFS не находит ни одного ребра → `painted: 0`.

Для реверс-only сетей (например П.16) краска струи отсутствует. Это известное ограничение, **не критично** для расчёта, но визуально юзер может ждать раскраски.

**Возможный фикс** (если попросят): дополнительный BFS для реверс-фанов через `inEdges`, чтобы покрасить путь от атмосферы/dead-end к фану. Не делал — заказчик не просил.

#### Закрытый контур без атмосферы / dead-end
В `airSolver.js` `solveNodePotentials` фан трактуется как «pressure boundary», а не «hard flow constraint». Если в сети нет ни одной атмосферы / dead-end и фан в замкнутом контуре, солвер не падает, но даёт нефизичные Q (Q фана может не совпадать с конфигурированным Q0). Это особенность модели Гаусс-Зейделя на потенциалах, **не баг моих правок**.
