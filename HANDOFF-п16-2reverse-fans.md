## Где зона правки (защищённый блок)

**Файл:** [js-for-index4/airSolver.js](js-for-index4/airSolver.js)

**Строки 486-571** — алгоритм ориентации рёбер (3 шага):
- **A (502-512)** — берём нарисованное направление как базовое.
- **B (514-552)** — итеративный flip перевёрнутых линий через BFS от источников.
- **Несвязные подграфы (558-571)** — ориентация от не-тупика.

**Подозреваемая зона:**
[airSolver.js:531-535](js-for-index4/airSolver.js#L531) — BFS считает источником ЛЮБОЙ узел с `node.sourceFlow !== 0` (включая отрицательное от reverse-вентилятора), а также узлы с `inDegree === 0`:

```js
nodes.forEach((node, key) => {
  if (node.sourceFlow !== 0 || inDegB.get(key) === 0) {
    reachableB.add(key); bfsB.push(key);
  }
});
```

`sourceFlow` берётся из `getObjectSupplyContribution()` ([airModel.js:75-86](js-for-index4/airModel.js#L75)) — для reverse-вентилятора возвращает `-vol`.

**Гипотеза:** для двух reverse-фанов «sources» BFS = оба этих узла. Алгоритм пытается достичь от них всех остальных узлов. Если на пути встречается ребро, у которого `fromKey` достижим из обоих, а `toKey` тоже достижим — flip не делается. Если `fromKey` достижим только из fan #1, но `toKey` достижим из fan #2 — текущая проверка `!reachableB.has(dir.fromKey) && reachableB.has(dir.toKey)` не сработает (оба «достижимы» в одном `reachableB`). В итоге часть рёбер остаётся неправильно ориентированной.

Вторая возможная гипотеза: `sourceFlow !== 0` включает и положительные, и отрицательные источники без различения. Reverse-фан физически — сток (inlet), а BFS-from-source трактует его как outlet.

## Параллельная защитная зона

[airModel.js:75-86](js-for-index4/airModel.js#L75) — `getObjectSupplyContribution()`:
```js
const vol = Math.abs(roundTo5(parseFloat(props.airVolume) || 0));
return (props.fanMode === 'reverse') ? -vol : vol;
```

`Math.abs` добавлен в этой сессии (п.17), чтобы устранить «двойной отрицательный» баг (Q=-100 + reverse → раньше давало +100). После фикса reverse-фан **гарантированно** возвращает отрицательный sourceFlow. Это **могло** изменить поведение при двух reverse-фанах — нужно проверить, не зависел ли защищённый блок от старого (нормализованного по знаку) поведения.

- Расходы получаются нулевые.
- Расходы получаются с неправильным знаком.

### Шаг 3. Диагностика BFS-ориентации
Добавить временные `console.log` в [airSolver.js:514-552](js-for-index4/airSolver.js#L514):
- `sourceNodeKeys` — кто считается источником.
- `node.sourceFlow` для каждого узла источника.
- На каждой итерации B: `reachableB`, какие линии переворачиваются.

Это покажет, что именно идёт не так.

### Шаг 4. Фикс (варианты)
Возможные подходы (выбрать после диагностики):
- **A.** Различать источники по знаку: использовать только узлы с `sourceFlow > 0` как «отправителей» BFS, узлы с `sourceFlow < 0` — как «получателей» (отдельный BFS вспять).
- **B.** Двунаправленный BFS: от positive sources вперёд, от negative sources назад; flip когда направление противоречит обоим.
- **C.** Pre-validation: запретить две ориентации навстречу в одной связной подсети (сообщить пользователю «измените режим одного из фанов»).
- **D.** Если 2 reverse фана — равноценны (зеркальная схема), их можно эквивалентно заменить на 2 supply фана с инвертированными координатами. Это математически чище, но потребует переосмыслить семантику reverse в общем случае.

Подход **A** — самый локальный и минимальный. Подход **C** — самый безопасный (избегает правки солвера, выводит warning).

### Шаг 5. Проверка

## Результаты сессии 2026-04-26 (продолжение по этому хендофу)

### Что сделано
1. **Шаг 1 — воспроизведение.** Загружен П.16.vnet через preview_eval (проп. helper `__loadVnet`). 7 линий, 2 реверс-фана @ Q=100, тупик 1160_160. Расчёт выполнен.
2. **Шаг 2 — эталон.** xlsx-эталона нет (`Математика к атомосфере (1).xlsx` не присутствует в проекте). Эталон построен вручную: при двух реверс-фанах + тупик → тупик играет роль implicit-атмосферы (degree-1 sink node), солвер должен дать Q=100 на обоих фанах, mass-balance по всем узлам.
3. **Шаг 3 — диагностика.** Добавлены guarded console.log в Шаг B BFS-ориентации (под флагом `window.__DEBUG_BFS`). Диагностика подтвердила:
   - reachable стартует с {фан1, фан2} (через `sourceFlow !== 0`) + 300_160 (inDeg=0, т.к. u6s1led9 нарисован 300→460)
   - BFS из 300_160 (имеет outEdge) достигает ВСЕХ 7 узлов
   - 300_340 не имеет outEdges (o79cy01r нарисован к фан2), но это не блокирует
   - **Шаг B не делает ни одного flip** — все направления остаются как нарисованные
   - Все 5 «перевёрнутых» направлений в финальном результате — это работа `solveNodePotentials` (Gauss-Zeidel + Q<0 reorientation, строки 186-197), **за пределами защищённого блока**
4. **Шаг 4 — решение по фиксу.** **Защищённый блок не модифицирован.** Диагностика опровергла обе гипотезы из исходного хендофа:
   - "BFS трактует оба реверс-фана как один reachable-набор" — да, оба попадают, но это не приводит к ошибочному flip (т.к. fan1 уже всё достигает)
   - "Reverse-фан как outlet вместо inlet" — формально да в `sourceFlow !== 0`, но это ускоряет схождение BFS, а не ломает
   - Временные логи откачены (см. diff airSolver.js — никаких остаточных правок)
5. **Шаг 5 — регресс.** Прогнаны все указанные тестовые файлы:

   | Файл | Линий | totalSourceFlow | Hсети, Pa | finite |
   |---|---|---|---|---|
   | П.8 | 6 | 0 | 0 | ✓ |
   | П.12 | 6 | -100 | 29.21 | ✓ |
   | П.13 | 6 | -100 | 29.21 | ✓ |
   | П.14 | 6 | -100 | 1.27 | ✓ |
   | П.16 | 7 | -200 | 75.53 | ✓ |
   | П.21 | 7 | +100 | 7.62 | ✓ |
   | П.24 | 21 | -200 | 29.16 | ✓ |

   Для П.16: оба фана получают Q≈100 как сконфигурировано (fan1=99.99999, fan2=100.00). Mass-balance корректен на всех узлах. Все стрелки направления (`createOrUpdateAirVolumeText`) визуально указывают по факту потока.

### Наблюдения вне зоны правки

Диагностика показала несколько мест, где **не-защищённый код может выдавать неожиданное поведение**, но это не относится к BFS-ориентации:

1. **Закрытый контур без implicit-атмосферы** ([airSolver.js:98-128](js-for-index4/airSolver.js#L98), `solveNodePotentials`). Если в сети нет тупика и нет атмосферы (только реверс-фаны на замкнутом контуре), солвер **не падает**, но даёт нефизичные Q (фан1 получил 33.33 вм. 100, фан2 — 133.33 вм. 100 в эксперименте с удалённой ветвью 1160). Это потому что Q фана трактуется как pressure-смещение в Gauss-Seidel, а не как hard-constraint. **Не относится к П.16** (там тупик есть), но потенциально объясняет жалобы про "ломает" в других сценариях.
2. **`applyFlowColoring`** ([visualization.js:367-432](js-for-index4/visualization.js#L367)) для реверс-фанов: BFS идёт по `outEdges`, но у реверс-фана после flip их нет — раскраска не применяется ни к одной линии. Для П.16 раскраска не показывает ничего (все фаны реверс). Возможно, пользователь этого ждёт.
3. **`totalSourceFlow = -200`** показывается в UI с лейблом «Подача: -200 м³/с» ([chainAnalysis.js:180](js-for-index4/chainAnalysis.js#L180)). Семантически "Подача" положителен по конвенции; для реверс-only сетей значение отрицательно — UX неоднозначен. Не баг расчёта.

### Рекомендация
**Перед любой правкой защищённой зоны — уточнить у пользователя точные шаги воспроизведения «ломает».** Тестовый файл `П.16.vnet` в текущем состоянии даёт численно корректный, mass-сбалансированный результат с правильными визуальными стрелками. Гипотезы из этого хендофа после диагностики не подтвердились.

Возможные направления (вне защищённого блока), которые могут быть истинной причиной жалоб из `Замечания.docx` («Расчёт правильный, направления беда»):
- Раскраска струи (`applyFlowColoring`) — ничего не красит для реверс-only сетей. Можно расширить, чтобы для сетей без supply-фанов BFS стартовал от dead-end-as-atmosphere узлов и красил по направлению потока к реверс-фанам.
- Лейбл "Подача" → "Расход через источники" + abs(value) для UX.
- Pre-validation warning: если в сети только реверс-фаны и нет dead-end / атмосферы — предупредить пользователя.
